# Observability-Driven Development (ODD)

Treat visibility as a first-class requirement. If it isn't observed, it doesn't exist in production.

## When to Use
- **Implementing New Features:** Every new entry point or core logic block must be instrumented.
- **Refactoring:** When changing complex logic, ensure existing observability is preserved or improved.
- **Debugging:** When a bug is hard to find, use this skill to "light up" the dark corners of the code.

## The ODD Process

### Phase 1: Tracer & Span Identification
1.  **Identify the Boundary:** Determine the start and end of the operation you are instrumenting.
2.  **Define the Span:** Give it a clear, semantic name (e.g., `checkout.process_payment`).
3.  **Trace Propagation:** Ensure the parent `trace_id` is passed into this new span.

### Phase 2: Instrumentation
1.  **Select Pattern:** Consult [instrumentation_patterns.md](references/instrumentation_patterns.md) for the appropriate boilerplate for the project's language (Python, Go, JS, Java, etc.).
2.  **Apply Local Patterns:** Adapt the conceptual boilerplate to the project's specific logging/tracing libraries (e.g., Pino, Winston, structlog, slog, OpenTelemetry SDK).
3.  **Audit Context:** Add relevant attributes from [otel_conventions.md](references/otel_conventions.md).

### Phase 3: Structured Logging
1.  **Enforce Schema:** Ensure all logs follow the [ecs_schema.md](references/ecs_schema.md).
2.  **Static Messages:** Keep `message` fields constant. Use the `attributes` object for dynamic values (IDs, amounts, etc.).
3.  **Failure Modes:** Every `catch` block MUST log structured error data.

### Phase 4: Verification
1.  **Log Audit:** Print a sample JSON log generated by the new code.
2.  **Alerting Draft:** (Optional) Draft a Prometheus/CloudWatch alerting rule that would trigger if this code fails.

## Rationalization Rebuttals

| Agent Rationalization | Rebuttal |
| :--- | :--- |
| "Logging slows down the hot path." | "Structured logging with a high-performance library (like Pino) has negligible overhead. Blindly running without logs is a higher risk." |
| "The framework already logs errors." | "Framework logs lack business context. We need to know *which* user/order failed, not just that a 500 occurred." |
| "I'll add monitoring later." | "Observability is not an afterthought; it is part of the implementation. A feature is not 'Done' until it is observable." |

## Red Flags
- 🚩 **Plain Text Logs:** Using `console.log` or interpolated strings like `logger.info("User ${id} failed")`.
- 🚩 **Swallowing Errors:** Catching an exception without logging the stack trace or incrementing an error metric.
- 🚩 **Missing Trace IDs:** Operations that cannot be correlated across service boundaries.
- 🚩 **PII in Logs:** Logging emails, passwords, or credit card numbers without masking.
